# 0006. 型共有を common パッケージで実現する

## ステータス

承認済み

## 文脈

API のレスポンス型をフロントが直接消費する構成のため、両者で型のずれが起きると即座にバグになる。Python + TS のような言語横断構成では OpenAPI 等のスキーマ言語を経由する必要があったが、本プロジェクトはどちらも TypeScript なので TS の型定義をそのまま共有できる。

補遺 (2026-04-07): 本 ADR で定めた「common パッケージに型を置く」方針は維持しつつ、api パッケージのレイヤ構造を整理するにあたり、common に置くのを **front ↔ api 間の DTO 型のみ** に限定する方針を追加した。api 内部のドメインモデルは `packages/api/src/domain/` に別定義する。詳細は [ADR 0011](0011-api-layering.md) を参照。

候補:

- API レスポンス型を API 側で定義し、フロント側に手で再定義する
- OpenAPI を介して両側で型生成
- tRPC 等の RPC ツールキット
- 共有パッケージに型のみを置き、両側から `import` する

## 決定

`packages/common` パッケージに型のみを配置し、API / フロント双方から `@git-web/common` として import する。

具体方針:

- common には実行時コードを置かない（純粋に `.d.ts` 相当の型定義のみ）
- API ハンドラは戻り値型を `@git-web/common` の型でアノテートする
- フロントは `fetch` の結果を `@git-web/common` の型でキャストする
- ランタイム検証（zod など）は当面入れない。必要になった段階で別 ADR

## 結果

- 型のずれは TypeScript コンパイラが検出する
- ランタイムの境界（fetch の結果）には型保証が無いため、API バグでフロントが壊れる可能性は残る。リスクが顕在化したら zod 導入を検討
- `tsc -b` のプロジェクトリファレンスで common → api / front の依存解決を行う
