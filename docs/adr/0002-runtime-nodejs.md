# 0002. ランタイムに Node.js を採用する

## ステータス

承認済み

## 文脈

git-web は API + フロントエンドの構成を取る。フロントエンドは TypeScript で書くことが先に決まっている。API 側のランタイム選定にあたり、以下の候補を検討した:

- Go + go-git: 単一バイナリ配布が容易
- Rust + gitoxide / git2: 高性能
- Node.js + TypeScript: フロントと言語統一
- Python + 軽量フレームワーク: 書きやすさ

評価軸:

- フロントとの型共有のしやすさ
- ローカルツールとしての起動の取り回し
- 学習・実装コスト
- 配布形態（今回は配布を考えないため重みは低い）

## 決定

Node.js を採用する。

理由:

- フロントが TS なので、API も TS にすることで型定義を `common` パッケージ経由で共有できる
- ランタイムが 1 つで済む（Python と Node 両方インストールが不要）
- `node:http` という stdlib に薄い HTTP モジュールがあり、wsgiref 並みの薄さで API を書ける
- 配布を考慮しないため、Go の単一バイナリの旨味は今回不要
- `bin/git-web` を `#!/usr/bin/env node` のシェバングスクリプトにすれば `git web` 経由の起動も自然

## 結果

- フロント / API の型を共通の TS 定義から導出できる
- Node.js ランタイムへの依存が前提となる
- 配布したくなった場合は別途バンドル戦略（pkg, ncc 等）を検討する必要がある
