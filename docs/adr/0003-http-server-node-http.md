# 0003. API HTTP サーバーに `node:http` + 自前ルーターを採用する

## ステータス

承認済み

## 文脈

API は git の状態を JSON で返すだけのごく薄いレイヤであり、フレームワーク機能（ORM、テンプレートエンジン、認証、セッション、複雑なミドルウェア等）はほぼ不要。
ユーザーから「WSGI くらい薄いものが好み」という要望があった。

候補:

- Express: 枯れているが歴史的しがらみとミドルウェア前提
- Fastify: 堅実で速い、ただし機能は今回過剰
- Hono: モダンで型強い、薄い。ただし依存が増える
- `node:http` (stdlib) + 自前ルーター: 依存ゼロ、最薄

## 決定

`node:http` を直接使う。ルーティングは自前の極小実装（`Map<\`${method} ${path}\`, handler>` ベース、必要なら正規表現での動的パラメータ）で対応する。

理由:

- wsgiref 相当の薄さという要件にもっとも合致
- 依存ゼロでセキュリティ・メンテ負荷が最小
- ローカル単独利用なので並行性は Node のイベントループで十分（追加の対応は不要）
- ルーター自体が小さいためテストしやすく、責務も明確

## 結果

- 依存最小、攻撃面最小
- ボディパース、CORS、エラーハンドリングなどは自前で書く必要があるが、ローカル単独・read-only 主体ではいずれも軽微
- 将来 API が肥大化した場合は Hono への移行を検討する余地を残す
- ルーターは純粋関数として実装し、単体テストで動作を担保する
