# 0009. セキュリティ境界とエンドポイント設計の禁則

## ステータス

承認済み

## 補遺

- ADR 0018: §1 エンドポイント一覧の `/api/refs` を実装済みとし、§2 のリビジョン許可文字集合をブランチ名・タグ名・`HEAD^^` 等を含む形に拡張 (本 ADR の当時の記述はそのまま残す)
- ADR 0059: §5 が予告していた状態変更系 (書き込み) エンドポイントを具体化。POST body 読み取り経路・Origin ガード (POST のみ・fail-closed・3 表記許容)・reviewsDir のパストラバーサル防御を定義 (本 ADR の当時の記述はそのまま残す)

## 文脈

git-web は API 経由で git CLI を実行する。設計を間違えると、ローカルツールであっても以下のような脆弱性を抱えうる:

- ブラウザ経由で任意の git サブコマンドを実行可能になる
- リポジトリ外のパスにアクセスできる
- 悪意あるサイトが iframe / fetch 経由で API を叩いて操作できる
- リダイレクトを伴うエンドポイントが踏み台になる

ローカル単独利用前提でも、ブラウザを通じてアクセスする以上、悪意のある web ページから 127.0.0.1 への CSRF / DNS rebinding 等の経路は現実に存在する。

## 決定

API のエンドポイント設計と git 呼び出しには以下の禁則を課す。

### 1. 汎用 git ランナー型エンドポイントを作らない

`POST /api/git`（body にコマンド名を含む）のような、**クライアントから git のサブコマンド名や引数を受け取って実行するエンドポイントを作ってはいけない**。代わりに、用途ごとに固有のエンドポイントを作る:

- `GET /api/log` → 内部で `git log` を叩く
- `GET /api/diff?from=...&to=...` → 内部で `git diff` を叩く
- `GET /api/refs` → 内部で `git for-each-ref` を叩く
- `GET /api/blob/:rev/:path` → 内部で `git cat-file` を叩く

各エンドポイントが叩くサブコマンドはサーバー側コードに**ハードコード**する。

### 2. クライアントからの入力はバリデーション後にのみ git に渡す

エンドポイントが受け取る `from` `to` `path` 等は、サーバー側で形式チェックしてから git に渡す:

- リビジョン文字列: `[0-9a-f]{4,40}` または ref 名の安全文字集合のみ許可
- パス: `..` を含まない、絶対パスでない、リポジトリルートからの相対パスとして正規化後にスコープ確認
- 任意文字列を git の引数末尾に置く際は `--` 区切りを徹底

### 3. シェル経由実行禁止

`child_process.execFile` のみを使い、`exec` / `spawn({ shell: true })` は禁止。引数は必ず配列で渡す。

### 4. 対象リポジトリは起動時 cwd 固定

クライアントから対象リポジトリのパスを受け取らない。起動時の `process.cwd()` を `git rev-parse --show-toplevel` で正規化して固定する。`--git-dir` / `--work-tree` を引数で受け付けない。

### 5. CORS / Origin 制御

- API は `127.0.0.1` のみに bind し `0.0.0.0` には bind しない
- すべての API レスポンスに `Access-Control-Allow-Origin` を付けない（同一オリジンのみ）
- 状態変更系（後続フェーズで `git add` 等を実装する場合）は `Origin` ヘッダの厳格チェックを行い、起動時に発行した自身のオリジン以外は 403
- DNS rebinding 対策として `Host` ヘッダも検査し、`127.0.0.1:<port>` か `localhost:<port>` 以外は 403

### 6. 静的ファイル配信のパストラバーサル対策

`packages/front/dist` 配下を配信する際、要求パスを `path.resolve` してから dist ディレクトリの prefix 配下にあることを確認する。symlink 越境は拒否する。

## 結果

- API の表面が「やってよいこと」だけに限定される
- 新しい git 操作を増やすたびにエンドポイントを追加するコストはかかるが、それは設計上の安全弁として受け入れる
- 状態変更系を実装するフェーズで本 ADR の Origin / CSRF 対策節を再確認する
