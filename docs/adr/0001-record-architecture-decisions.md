# 0001. アーキテクチャ決定記録 (ADR) を残す

## ステータス

承認済み

## 文脈

git-web は個人ローカルツールではあるが、フロント FW、API ランタイム、ビルド構成、テスト基盤など複数の技術選定が必要であり、選定理由を後から追えるようにしておきたい。判断の根拠を口頭やコミットメッセージに散らすと、後日「なぜこの構成にしたのか」を再構成するコストが高い。

## 決定

Michael Nygard 形式の簡易版 ADR を `docs/adr/` に配置する。
ファイル名は `NNNN-kebab-case-title.md`（4桁ゼロパディング、連番）。
各 ADR は以下のセクションを持つ:

- ステータス（提案中 / 承認済み / 廃止 / 置換）
- 文脈
- 決定
- 結果

決定が変わった場合は新しい ADR を起票する。旧 ADR の本文（決定・結果）は当時の判断として書き換えず、文脈セクションに「変更があった新 ADR へのリンク」のみ追記する。詳細な運用は `.claude/rules/dev-process.md` を参照。

## 結果

- 設計判断の経緯が永続的に追跡可能になる
- ADR を書く一手間が増えるが、判断のたびにコミットメッセージで長文を書くより構造化される
- 新しい貢献者（将来の自分含む）が前提を素早く把握できる
