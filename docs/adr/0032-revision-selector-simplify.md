# ADR 0032: リビジョンセレクタの表示順変更と件数制限撤廃

## ステータス

承認済み

## コンテキスト

ADR 0025 で定めたリビジョンセレクタの候補リストには以下の課題がある。

- `head`（チェックアウト中のブランチ名）を HEAD 直後に優先表示しているが、ブランチ一覧にも含まれるため冗長である
- `limit` による件数制限があり、ブランチ数が多い場合に候補が切り詰められる。切り詰めが発生するとユーザーが目的のブランチを見つけにくい
- `truncated` フラグは UI 上で活用されておらず、情報として不要である

本 ADR は ADR 0025 の決定事項を一部変更する。

## 決定事項

### 表示順の変更

RevisionCombobox の `options` computed における候補の並び順を以下に変更する。

1. `(worktree)` — `allowWorktree` が true の場合のみ
2. `defaultBranch` — null でなければ表示
3. `HEAD` — 文字列リテラル
4. 残りの branches（重複排除済み、件数制限なし）
5. tags（重複排除済み、件数制限なし）

ADR 0025 にあった `head`（チェックアウト中のブランチ名）の優先枠は削除する。

### 件数制限の撤廃

- バックエンドの `refs` API からブランチ・タグの件数制限（`limit` パラメータ）を撤廃する
- `limit` クエリパラメータが送信された場合は無視する（後方互換のためエラーにしない）
- `RefListDto` / `RefList` から `truncated` フィールドを削除する

### head フィールドの削除

- `RefListDto` / `RefList` から `head` フィールドを削除する
- バックエンドの `GitRefsClient.headRef()` は使用箇所がなくなるため port / adapter からも削除する

### コミットハッシュ検索について

コミットハッシュの候補リスト検索は行わない。`git rev-list --all` による全コミット列挙は O(N)（N = コミット数）であり、リクエスト毎に走査が必要になる。ブランチ・タグ数（O(B+T)、B+T << N）と比較してオーダーが大きく異なるため、候補リスト検索の対象外とする。

コミットハッシュの直接入力（自由入力）は従来通り受け付ける。`parseRevision` で `[0-9a-f]{4,40}` が許可されており、リストに表示されなくても入力・確定は可能である。

## 根拠

- `head` の優先枠はブランチ一覧との重複排除で実質的に意味がないケースが多い
- 件数制限の撤廃により、ブランチ数が多いリポジトリでも全候補にアクセスできる
- ローカル Git リポジトリを対象とするツールであり、レスポンスサイズの増加（数千ブランチで数百 KB 程度）は許容範囲と判断する
- `limit` / `truncated` の削除により API とフロントエンドの実装が簡素化される
