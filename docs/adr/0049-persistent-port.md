# ADR 0049: レジストリによるリポジトリ別ポート永続化

## ステータス

承認

## コンテキスト

ADR 0044 でデフォルトポートを OS 自動割当（port 0）に変更し、インスタンスレジストリを導入した。レジストリにはリポジトリごとの起動情報（port, pid, url, startedAt）が記録される。

しかし、プロセス終了時に unregister でエントリを削除し、加えて起動時に pruneStale で stale エントリを除去するため、再起動のたびに異なるポートが割り当てられる。ブックマークや開発ツールの設定など、リポジトリごとにポートが固定されていると便利な場面がある。

レジストリには既にリポジトリとポートの対応が記録されている。エントリを削除せず残せば、ポートの永続化は自然に実現できる。stale かどうかの判定は `isEntryLive` で都度行えばよく、エントリの削除は不要である。

## 決定

### エントリを削除しない

- `pruneStale`（stale エントリの一括除去）を廃止する
- `removeEntry`（単一エントリの除���）を廃止する
- `unregister` / `unregisterSync`（プロセス終了時のエントリ削除）を廃止する
- レジストリは追記・上書きのみ行い、エントリの削除は行わない

### 起動フローの変更

1. レジストリをロードし、自リポジト��のエントリを検索する
2. **エントリあり + live** → 既存インスタンスの URL を返す（従来通り）
3. **エントリあり + stale** → そのエントリのポートで `start()` を呼ぶ。EADDRINUSE の場��は port=0 でフォールバック起動する
4. **エ��トリなし**（初回起動）→ port=0 で OS に割当を任せる。割り当てられたポートがレジストリ内の他エントリのポートと重複していた場合は、サーバーを閉じて port=0 で再起動する（最大 1 回）
5. 起動成功後、レジストリにエントリを upsert する

### start() の listen 失敗���のリソース解放

`start()` 内で `listen()` が失敗した場合、作成済みの HTTP サーバーオブジェクトを `close()` してからエラーを再送出する。これにより EADDRINUSE フォールバック時のリソースリークを防ぐ。

### ポート重複の回避

新規起動（エントリなし）で OS に割り当てられたポートが、レジストリ内の他リポジトリのエントリと同じポートだった場合、そのリポジトリが次回起動時に EADDRINUSE で不要なフォールバックを強いられる。これを避けるため、新規起動時はレジスト��内の既存ポート一覧と照合し、重複していたら 1 回だけ���トライする。

## 根拠

- レジストリにはリ���ジトリとポートのマッピングが既に記録されている。エントリを削除しないだけでポート永続化が実現できる
- live/stale の判定は `isEntryLive`（PID 確認 + HTTP ヘルスチェック）で毎回行うため、エントリの存在 = プロセスの生存ではない。エントリ削除に頼る必要がない
- 個人の開発マシンで扱うリポジトリ数は多くても数十程度であり、エントリを削除しないことによるレジストリファイルの肥大化は実用上問題にならない

## 影響

- リポジトリごとに一度割り当てられたポートが再起動後も維持される
- `PORT` 環境変数による明示指定は引き続き最優先で機能する
- bin/git-web の SIGINT/SIGTERM ハンドラが簡素化される（unregister 不要）
- `process.on('exit')` の保険経路も不要になる
- `readRegistrySync` / `writeRegistrySync`（同期版レジストリ操作）が��要になる
