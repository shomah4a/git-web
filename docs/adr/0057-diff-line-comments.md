# 0057. diff view への行コメント (レビュー機能)

## ステータス

承認

## 文脈

coding agent 駆動開発では生成される diff が多く、コードレベルのレビュー負荷が高い。
GitHub の PR レビューのように **diff の行へコメントを付け**、それをリポジトリ内のファイルに
保存して外部 agent (Claude) が読んで対応する、というワークフローを実現したい。

git-web は現状 read-only (GET のみ) であり、コメントの保存には初の状態変更系 (書き込み) を
導入する必要がある。本 ADR は機能スコープとアンカー定義・エンドポイントを定める。
永続化フォーマットは ADR 0058、書き込みのセキュリティ境界は ADR 0059、表示時の行翻訳は
ADR 0060 で扱う。

## 決定

### 1. アンカー定義 (コメントが指す対象)

コメントは以下の三つ組でアンカーする:

- **コミット SHA (40桁)**: コメントを付けた時点の `to` リビジョンの完全な commit SHA
- **path**: リポジトリルートからの相対パス
- **new 側の連続行範囲** (`newLineStart` 〜 `newLineEnd`, 1-based, inclusive)

理由: GitHub の RIGHT side コメントと同型。diff の new 側 (追加・文脈行) のみを対象とし、
削除行 (old 側) にはコメントを付けない。範囲は連続行のみ (飛び地は持たない)。

### 2. 作成可能条件 (to が具体コミットのときのみ)

コメント作成は **`to` が具体的なコミット (40桁 SHA に解決できる ref/SHA) のときのみ** 許可する。
`to=作業ツリー` (worktree) は commit SHA を持たないためコメント作成不可とする。

- アンカーには ref 文字列ではなく解決済みの 40桁 SHA を保存する (ref は動くため)
- server 側で「`to` が 40桁 SHA でなければ 400」を必須とする (fail-closed)。front の
  作成 UI 非表示は UX 補助であり、検証の真実は server 側に置く

### 3. スレッドはフラット

返信スレッドは持たない。1 アンカーに対し独立したコメントが 0..n 件並ぶ。
将来スレッド化する場合も本 ADR のフラットモデルを壊さない追加で行う (YAGNI)。

### 4. resolved は UI からのみ操作

コメントの「対応済み」状態 (resolved) は **git-web の UI からのみ** トグルする。
外部 agent (Claude) はコメントを **読むだけ** で、resolved を操作しない。

### 5. エンドポイント (用途固有・ハードコード, ADR 0009 §1 準拠)

汎用ランナーは作らず、用途ごとに固有のエンドポイントを置く。

| メソッド | パス                      | 用途                       |
| -------- | ------------------------- | -------------------------- |
| GET      | `/api/reviews?sha=<40桁>` | 指定コミットのコメント一覧 |
| POST     | `/api/reviews`            | コメント作成               |
| POST     | `/api/reviews/resolve`    | resolved 切替              |

- ルーティングは既存の method+path の exact match (動的パスパラメータなし) を踏襲するため、
  対象 id 等は query / body で受け渡す
- POST は状態変更系のため ADR 0059 の Origin/Host ガードの対象とする

### 6. 表示時の突合 (概要)

diff view で表示中の `from..to` 範囲に含まれる各コミット SHA の review を読み、各コメントを
現在の `to` の行へ翻訳して表示する。翻訳の詳細とフォールバック (outdated) は ADR 0060。

### 7. 自己汚染の許容

保存先 `.git-web/reviews/` は repo 配下にあるため、`to=作業ツリー` の diff や tree view に
untracked ファイルとして出現しうる (自己汚染)。本機能ではこれを**許容**し、diff/tree 側の
フィルタは追加しない。実装の単純さを優先し、git 追跡設定 (.gitignore) には git-web は関知
しない (ADR 0058)。

## 非採用案

- **old 側 (削除行) へのコメント**: アンカーが from の SHA + oldLineNo になり二系統化する。
  MVP では new 側のみに絞る。
- **作業ツリーへのコメント**: commit SHA を持たずアンカーが不安定。HEAD 代用や特別キーは
  複雑化を招くため不採用。
- **スレッド (返信)**: フラットで足りる。複雑度増を避ける。

## 結果

- diff レビューを行レベルで行え、結果をリポジトリ内ファイルとして外部 agent が読める
- 書き込み系の導入により ADR 0009 §5 のセキュリティ境界を具体化する必要が生じる (ADR 0059)
- リネーム跨ぎの追従は対象外 (ADR 0060 の既知制約)
