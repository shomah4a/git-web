# ADR 0034: mode 表示を ls 形式に変換する

## ステータス

承認済み

## コンテキスト

ADR 0026 で RevisionTreeView に mode カラムを追加したが、表示は Git の 8 進数文字列（`100644`, `040000` 等）をそのまま出力している。
WorktreeView でも同様である。

この表示は Git 内部表現であり、ユーザーにとっては `ls -l` で馴染みのある `drwxr-xr-x` 形式の方が直感的に読みやすい。

## 決定事項

### ビット厳密パース方式の採用

mode の 8 進数文字列を `parseInt(mode, 8)` で数値に変換し、ビット演算で各フラグを判定して ls 形式文字列を組み立てる。
既知パターンのマッピングテーブルではなく、汎用的なビットパースとする。

### ファイルタイプの判定

8 進数 mode の上位ビット（POSIX ファイルタイプ）から先頭文字を決定する:

| ファイルタイプビット | 先頭文字 | 説明                    |
| -------------------- | -------- | ----------------------- |
| `0040000`            | `d`      | ディレクトリ            |
| `0120000`            | `l`      | シンボリックリンク      |
| `0160000`            | `m`      | Git submodule (gitlink) |
| その他               | `-`      | 通常ファイル等          |

Git の submodule (mode `160000`) は POSIX に対応するファイルタイプが存在しないため、`m` (module) を先頭文字とする。

### パーミッションビットの判定

下位 9 ビットを owner / group / other の 3 グループに分割し、各ビットに対応する `r`, `w`, `x` または `-` を出力する。

Git が返す mode のパーミッションビットの実態:

- `100644`: 下位 9 ビット = `110100100` → `rw-r--r--`
- `100755`: 下位 9 ビット = `111101101` → `rwxr-xr-x`
- `040000`: 下位 9 ビット = `000000000` → `---------`
- `120000`: 下位 9 ビット = `000000000` → `---------`
- `160000`: 下位 9 ビット = `000000000` → `---------`

Git の tree object (`040000`) やシンボリックリンク (`120000`) はパーミッションビットが常に 0 であるため、`d---------` や `l---------` と表示される。これは Git が実際に返す値のビット厳密な表現であり、`ls -l` コマンドがファイルシステムの実パーミッションを表示する `drwxr-xr-x` とは異なる。

POSIX の setuid / setgid / sticky ビットには対応しない。Git はこれらのビットを mode に含めないため、実害はない。

### フォールバック

`parseInt(mode, 8)` が `NaN` を返す場合（不正な入力）は、元の文字列をそのまま返す。

### 変更対象

- `packages/front/src/format/entry.ts` の `formatMode` 関数のみ
- 関数シグネチャは変更しない
- バックエンド・DTO は変更なし

## 根拠

- `ls -l` 形式はファイルパーミッションの表記として広く認知されており、8 進数文字列より直感的である
- ビット厳密パースにより、将来 Git が新たな mode 値を返した場合にも対応可能
- 変更が表示フォーマット関数 1 つに閉じており、影響範囲が限定的である
