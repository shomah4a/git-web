# パッケージ管理

## 背景

本ファイルのルール群は、近年顕在化している npm エコシステムへの supply chain attack（侵害されたパッケージの公開、悪意あるバージョンの押し付け、phantom dependency の悪用など）への対策を主目的とする。
個別のルールがどの攻撃面を塞ぐためのものかを意識して運用すること。

## パッケージマネージャ

本プロジェクトのパッケージマネージャは pnpm を使用する。`npm` / `yarn` コマンドは使わないこと。

### リポジトリローカルの pnpm を使うこと

システムに `pnpm` がインストールされていても、リポジトリ内の **`./bin/pnpm` ラッパー経由で呼び出すこと**。`corepack enable` のようなシステム環境を変更する操作は禁止する。

`./bin/pnpm` の動作:

- `package.json` の `packageManager` フィールドで指定された pnpm バージョンを corepack 経由で使用する
- corepack のキャッシュを `${REPO_ROOT}/.corepack/` に閉じ込め、システムやユーザーホームを汚さない
- リポジトリごとに pnpm のバージョンが独立する

理由（環境分離の観点）:

- システム / ユーザーグローバルの pnpm を使うと、他リポジトリやユーザー設定の影響を受けてビルドの再現性が落ちる
- システムを変更する操作（`corepack enable` 等）は「1 リポジトリの分際」を超えており、ユーザー環境への副作用として不適切
- リポジトリローカルに閉じ込めることで、`packageManager` フィールドに書かれたバージョンが事実上の真実になる

### 使用するコマンド例

- 依存追加: `./bin/pnpm add` （`npm install` ではない）
- 依存削除: `./bin/pnpm remove`
- スクリプト実行: `./bin/pnpm <script>` または `./bin/pnpm run <script>`
- workspace 横断実行: `./bin/pnpm -r <script>` / `./bin/pnpm --filter <pkg> <script>`
- ロックファイルは `pnpm-lock.yaml` のみコミットする（`package-lock.json` / `yarn.lock` は作らない）

### pnpm を使う理由（supply chain 観点）

- pnpm は依存をシンボリックリンクで隔離するため、`package.json` に書かれていない phantom dependency にコードがアクセスできない。これにより、間接依存の侵害が直接利用に化けるリスクが減る
- `pnpm-lock.yaml` は各パッケージの integrity ハッシュを記録しており、改ざんを検知できる
- 複数のパッケージマネージャが混在するとロックファイルの整合性が担保できなくなるため、1 種類に統一する

## バージョン固定（追加時）

npm パッケージを新規追加する際は、追加前に最新安定版を調べたうえで、完全バージョン指定で固定すること。

### 手順

- `pnpm view <パッケージ名> version` で最新安定版を確認する
- `pnpm add <パッケージ名>@<バージョン>` のように完全バージョン指定で追加する
- `package.json` に `^` / `~` などのプレフィックスを残さない
- これを機械的に強制するため、リポジトリの `.npmrc` に `save-prefix=""` を設定する

### バージョン固定の理由（supply chain 観点）

- `^1.2.3` のようなレンジ指定は、後から公開された不正バージョンを `pnpm install` 時に自動で取得してしまう可能性がある
- ロックファイルがあれば一度固定されるが、新規環境への初回 install や更新作業のタイミングで穴が生じる
- 完全固定にしておけば、バージョンを上げるのは常に明示的な作業（または dependabot のような追跡可能な経路）になる

## 更新時

依存パッケージの更新は dependabot 等の自動化ツールに任せる方針とする（後続フェーズで導入予定）。
本ルールはあくまで「追加時の初期バージョン固定」のみを対象とする。
