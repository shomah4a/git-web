#### 決定事項 (ADR 0016)

- エンドポイント: `GET /api/blob?path=<path>&rev=<rev>`
  - rev キーなし = worktree (null)
  - rev=<値> = 指定リビジョンを `git cat-file blob` で取得
  - rev= (空文字) = 400 (parseRevision で失敗させる)
- `git show` ではなく `git cat-file blob` を採用。理由: `git show HEAD:<dir>` が tree listing を 200 で返す事故を設計時点で回避するため (ディレクトリ指定でも cat-file blob は非ゼロ終了して 404 になる)
- バイナリ判定は NUL バイトの存在。binary 時は `content: ''`
- サイズ上限 / truncated は設けない (ローカル動作前提、front 側でガード予定)
- realpath ベースの repo root 境界検査 (純粋関数 `isInsideRepo` に切り出し、TOCTOU は許容)
- `parseDiffPath` の `..` 検査を **部分一致 → segment ベース** に修正 (`foo..bar` のような正規ファイル名を通すため)
- HTTP レスポンスヘルパを `http/response.ts` に集約 (diff / blob / repo / error-mapper の重複解消)

## セッション中の教訓

- **計画承認前の ADR 起票を飛ばしかけた**: セッション開始直後に `parseDiffPath` の実装検討に進もうとして、ユーザから「ADR 書かないんですか」と指摘された。プロジェクトルール `dev-process.md` の「プランの最初に ADR を作ること」は厳密に守る必要がある
- **防衛評価 HIGH を真面目に反映すると事故が減る**: 着手前に `git show` の落とし穴 / realpath 境界検査の穴 / parseDiffPath の誤拒否を潰せたことで、実装安全性評価では HIGH/CRITICAL ゼロで済んだ
- **例外の握りつぶしは死罪**: `catch (err) { throw new OtherError() }` で元 err を捨てるのは原因追跡不能になる。`{ cause: err }` で保持する習慣を全ドメインエラーで徹底する。`Object.assign(err, ...)` で err を mutate するのも同罪 (呼び出し元からの参照が壊れる / 副作用追跡困難)。新規 Error クラスに wrap する
- **cat-file blob vs git show**: blob 取得は必ず `cat-file blob` を使う。`git show HEAD:<dir>` は tree listing を 200 で返す事故のもと
- **common パッケージの build を忘れない**: `BlobDto` を `common/src` に追加した直後に api 側で import エラー。`./bin/pnpm --filter @git-web/common build` を明示的に回す必要がある
